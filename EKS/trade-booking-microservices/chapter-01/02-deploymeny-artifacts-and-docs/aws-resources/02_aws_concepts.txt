The AWS Load Balancer Controller is a tool that automatically creates and manages AWS load balancers for your Kubernetes apps in EKS.

In simple terms:

You define how traffic should reach your app (using Ingress or Service in Kubernetes)
The controller automatically sets up the AWS load balancer (ALB/NLB)
It also keeps it updated when your app changes

👉 Think of it as a bridge between Kubernetes and AWS networking, so you don’t have to manually configure load balancers.
👉 It watches Kubernetes resources like Ingress and Service and creates AWS load balancers for you.


Request flow:
    We have:
    1. Domain → sandeep.com
    2. Service → refdata-service (port 80)
        Requirement:

        👉 Any request with /reference-data → should go to refdata-service

    Ingress Configuration
    Here’s what your Ingress should look like:
    apiVersion: networking.k8s.io/v1
    kind: Ingress
    metadata:
    name: refdata-ingress
    annotations:
        alb.ingress.kubernetes.io/scheme: internet-facing
    spec:
    rules:
        - host: sandeep.com
        http:
            paths:
            - path: /reference-data
                pathType: Prefix
                backend:
                service:
                    name: refdata-service
                    port:
                    number: 80

    A user hits http://sandeep.com/reference-data in their browser. The domain sandeep.com is already mapped (via DNS like Amazon Route 53) 
    to an AWS Application Load Balancer (ALB). This ALB was automatically created by the AWS Load Balancer Controller after it saw your Ingress 
    resource.

    When the request reaches the ALB, it checks two things: the host (sandeep.com) and the path (/reference-data). Based on your Ingress rule, 
    the controller has already configured the ALB with a listener rule that says: “If the path starts with /reference-data, forward traffic to 
    the target group for refdata-service.”

    The ALB then forwards the request to one of the targets (pods or nodes) inside your Amazon Elastic Kubernetes Service cluster. 
    Once inside the cluster, Kubernetes routes the request to the refdata-service, which load-balances it to one of your running pods.

    Your application processes the request and sends the response back through the same path: pod → service → ALB → user.
